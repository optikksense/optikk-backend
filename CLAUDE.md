## Quick reference

- **Stack**: Go 1.25, Gin, ClickHouse, MySQL, Redis, Kafka, OTLP gRPC (ingest surface only — self-telemetry is Prometheus-only).
- **Server entry**: `cmd/server/main.go`
- **Module registration**: `internal/app/server/modules_manifest.go` → `configuredModules()`.
- **Handler helpers**: `internal/shared/httputil/base.go` — `RespondOK`, `RespondErrorWithCause`, `ParseRequiredRange`.
- **Error codes**: `internal/shared/errorcode/codes.go`.
- **ClickHouse helpers**: `internal/infra/database/` — `QueryMaps`, `QueryCount`, `SqlTime`, type extractors.
- **Time bucketing**: `internal/infra/timebucket/timebucket.go` — single source of truth for every bucket value in the system. Spans/logs partition buckets (`SpansBucketStart`, `LogsBucketStart`), metrics tier buckets (`MetricsHourBucket` / `Metrics6HourBucket` / `MetricsDayBucket`, picked via `MetricsBucketAlign`), and display-time aggregation buckets (`DisplayBucket`). **No ClickHouse bucket functions** (`toStartOfHour`, `toStartOfInterval`, `toStartOfDay`, etc.) appear anywhere in our SQL — writer and reader compute identical values via the same Go helpers, so cross-language drift can't silently miss rows. Constants are compile-time fixed; changing them is a breaking schema change requiring a table rebuild (see grain-change runbook in the package doc).
- **Rollup Selection**: `internal/infra/rollup/tier.go` — `TierTableFor(prefix, startMs, endMs)` for smart table choice.
- **Session**: `internal/infra/session/manager.go`.
- **Middleware**: `internal/infra/middleware/` — public prefixes: `/api/v1/auth/login`, `/otlp/`, `/health`.
- **Ingestion**: `internal/ingestion/{spans,metrics,logs}/` — handler → mapper → kafka producer → dispatcher → per-partition worker → writer (CH batch + retry + DLQ). Shared generics in `internal/infra/kafka_ingest/` (`dispatcher.go`, `worker.go`, `writer.go`, `accumulator.go`, `metrics.go`, `pools.go`, `pipeline_cfg.go`). Tuning knobs live in `internal/config/ingestion.go` → `IngestPipelineConfig` (per-signal YAML overrides).
- **Local monitoring**: `monitoring/stack/docker-compose.yml` — Prometheus `:19091`, Grafana `:13001`. Dashboards in `monitoring/grafana/dashboards/`: `optikk_overview`, `optikk_http_api` (per-API drill-down), `optikk_grpc`, `optikk_db`, `optikk_redis`, `optikk_kafka`, `optikk_ingest`. All Prometheus-sourced; there is no OTel collector or Tempo — the `/metrics` endpoint is the only self-telemetry surface.
- **Load test (query-side)**: `make loadtest-smoke` for CI sanity, `make loadtest-all` for the full sweep. k6 scenarios live in `loadtest/scenarios/<module>/`; entrypoints in `loadtest/entrypoints/`. See `loadtest/docs/README.md` for the env-flag table and the Prometheus remote-write setup.
- **Schema migrations**: `db/clickhouse/*.sql` applied via `internal/infra/database_chmigrate`.
- **Query budgets**: `internal/infra/database/clickhouse.go` exposes three budgets — `Dashboard` (3s, sub-second panels), `Overview` (15s, infrastructure/saturation), `Explorer` (60s, ad-hoc). Use `DashboardCtx` / `OverviewCtx` / `ExplorerCtx` to attach.
- **HTTP metrics read path** (`internal/modules/services/httpmetrics/`): apm-style. Repository holds queries only; service holds derivation. 17 endpoints under `/api/v1/http/...` served by 6 repo methods, single canonical source per endpoint (no fallbacks). Histogram percentiles (request/body/client/dns/tls duration) come from `observability.metrics` via the apm `WITH active_fps AS (... metrics_resource ...)` CTE; P50/P95/P99 computed Go-side via `quantile.FromHistogram`. Route and external-host top-N panels (volume/latency/error rate) read `observability.spans` via a `WITH active_fps AS (... spans_resource ...)` CTE plus `ts_bucket` PREWHERE for granule pruning; P95 from ClickHouse SQL `quantile(0.95)(duration_nano)`.
- **Infrastructure read paths** (`internal/modules/infrastructure/`): apm-style across the board. `network` (8), `cpu` (7), `disk` (8), `memory` (6), `jvm` (7), `kubernetes` (9), and `resourceutil`'s direct query (1) all read `observability.metrics` via inline `WITH active_fps AS (SELECT fingerprint FROM observability.metrics_resource WHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd AND metric_name [= @metricName | IN @metricNames])` CTE — no `fmt.Sprintf`, no phantom-rollup column references (`value_sum`, `value_avg_num`, `sample_count`, `state_dim`), each query a `const query = \`...\`` string in its method body. OTel semconv 1.30+ canonical attribute paths (`attributes.'system.cpu.state'::String`, `attributes.'system.network.io.direction'::String`, `attributes.'jvm.memory.pool.name'::String`, `attributes.'k8s.container.name'::String`, etc.) — no `coalesce`/`nullIf` alias chains. `nodes` (3 methods) and `fleet` (1 method) stay on `observability.spans` for RED aggregates (`count()`, `countIf(has_error OR response_status_code >= 400)`, `quantile(0.95)(duration_nano)`) via `spans_resource` CTE — same pattern httpmetrics' route/external-host queries use. `connpool` was already correct on `observability.metrics`. **Repos hold queries only**; service holds derivation: counter→rate via `displaybucket.SumByTimeAndKey`, gauge avg via custom multi-key folds, multi-metric percentage blends (cpu 3-metric, memory 4-metric, disk 3-metric, connpool 5-metric), JVM GC duration P50/P95/P99 via `quantile.FromHistogram`, RED `error_rate`/`avg_latency_ms` derivation, `≤1.0 → *100` percentage normalization, NaN scrubbing. Each module owns its own `metricArgs` / `metricBucketBounds` / `withMetricName{,s}` / `spanArgs` / `spanBucketBounds` helpers locally — no cross-module shared helpers. `resourceutil` is a composite that consumes `*Service` (not `Repository`) from each domain submodule.
- **Kafka saturation read path** (`internal/modules/saturation/kafka/`): apm-style. 19 routes under `/api/v1/saturation/kafka/...` served by ~16 repo methods, **all metrics-only** (no spans). Every read goes through `observability.metrics` with a `WITH active_fps AS (SELECT fingerprint FROM observability.metrics_resource WHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd AND metric_name [IN @metricNames | = @metricName])` CTE; main query PREWHEREs on `(team_id, ts_bucket_hour, fingerprint IN active_fps)` to pin the first three slots of the metrics PK. Hour-aligned bucket bounds via `timebucket.MetricsHourBucket`. Topic / consumer_group / operation / partition / broker / error_type are extracted inline from `attributes JSON` typed-paths via `attrStringAny(...)` chain (each panel resolves multi-name OTel attribute aliases). **One CH query per repository method** (no `if`/`else` over two SQL strings, no fallbacks); panels needing multiple metrics (`summary-stats`, `e2e-latency`, `rebalance-signals`) compose multiple repo calls in `service.go`. Service derivations: counter rate via `displaybucket.SumByTime` ÷ `timebucket.DisplayGrain(window).Seconds()`, histogram percentiles via `quantile.FromHistogram(buckets, counts, q)` after merging aligned histograms per `(display_bucket, dim)` tuple in Go, gauge avg via custom multi-key folds, partition-lag snapshot via SQL `argMax(value, timestamp)` + DESC LIMIT 200. Histogram-counter `messaging.client.operation.duration` doubles as a fallback latency source: latency-by-topic / -by-group queries OR-match the canonical duration metric and the unified op-duration metric filtered by `messaging.operation.name IN @opAliases` to absorb older instrumentations. Error panels depend on OTel semconv 1.30+ `error.type` data-point attribute; older instrumentations leave them empty.
- **DB saturation read path** (`internal/modules/saturation/database/`): apm-style across all 9 submodules — `volume`, `errors`, `latency`, `collection`, `system` (single DB scoped), `systems` (cross-system list), `summary` (top-level KPIs), `slowqueries`, `connections` — plus `explorer` (no repo; composer over the others). Shared filter package at [internal/modules/saturation/database/filter/](internal/modules/saturation/database/filter/) owns `Filters` (DBSystem/Collection/Namespace/Server), `BuildSpanClauses(f)` + `BuildMetricClauses(f)`, `SpanArgs/SpanBucketBounds` (5-min) + `MetricArgs/MetricArgsMulti/MetricBucketBounds` (hour), `LatencyBucketBoundsMs` (`1/5/10/25/50/100/250/500/1000/2500/5000/10000/30000/+Inf` ms) and `LatencyBucketCountsSQL()` — the closed-set `[countIf(duration_nano …) …]` array emitter every latency-bearing query splices in. **Source-table choice per OTel semconv 1.30+**: 8 of 9 submodules read raw `observability.spans` with PREWHERE `(team_id, ts_bucket, fingerprint IN active_fps)`, accessing flat `db_system` / `db_statement` / `duration_nano` / `has_error` / `response_status_code` columns plus `attributes.'db.operation.name'/'db.collection.name'/'db.namespace'/'server.address'/'error.type'::String` typed-paths. The `connections` submodule alone reads `observability.metrics` (the `db.client.connection.*` family is instrumentation-side gauges/counters/histograms): `count` / `max` / `idle.max` / `idle.min` / `pending_requests` (gauges), `timeouts` (counter → rate per-second in service), `wait_time` / `create_time` / `use_time` (OTel native histograms with `hist_buckets` + `hist_counts` columns; service merges per `(display_bucket, pool.name)` and interpolates p50/p95/p99 via `quantile.FromHistogram`). Service.go owns all derivation: count→rate via `filter.BucketWidthSeconds`, error_count/total → ratio percent, gauge avg via custom multi-key folds, fixed-bucket histogram → percentile via `quantile.FromHistogram(filter.LatencyBucketBoundsMs, …)`, top-N sorting + trimming Go-side. **No phantom rollup table** — the previous `db_histograms_rollup` references (`latency_ms_digest`, `value_sum`, `sample_count`, `db_operation`, `pool_name`, `db_connection_state`, etc.) were all pointed at columns that never existed; every method now reads canonical columns + OTel typed-paths only. **No `fmt.Sprintf`, no `%s`** anywhere in the 9 repos. `internal/modules/saturation/database/internal/shared/` retains only HTTP-side helpers (`ParseFilters` / `ParseLimit` / `ParseThreshold` / `RequireCollection` / `RequireDBSystem` / `RegisterDualGET`); the previous `query.go` / `rollup.go` / `otel.go` — all of which assumed the phantom rollup — were deleted.
- **Logs read path**: apm-style across the four submodules — `explorer` (POST `/logs/query`, list + include orchestrator), `logdetail` (GET `/logs/:id`), `log_facets` (POST `/logs/facets`), `log_trends` (POST `/logs/trends`). Wire format is **flat typed JSON** — `{startTime, endTime, services:[], hosts:[], pods:[], containers:[], environments:[], severities:[], traceId, spanId, search, searchMode, attributes:[{key,op,value}], excludeServices:[], excludeHosts:[], excludeSeverities:[], …}` — no `[{field, op, value}]` parse step. The typed shape and clause emitter live in [internal/modules/logs/filter/filter.go](internal/modules/logs/filter/filter.go) (`filter.Filters` + `filter.AttrFilter` + `Validate()` for time clamp + searchMode default + `BuildClauses(f)`). Each filter-bearing repo method takes `filter.Filters` directly and calls **one shared `filter.BuildClauses(f) → (resourceWhere, where, args)`** that emits the inline `WITH active_fps AS (SELECT DISTINCT fingerprint FROM observability.logs_resource PREWHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd <resourceWhere>)` CTE fragment + the row-side WHERE fragment, both bound via `clickhouse.Named()`. Main scan PREWHEREs `(team_id, ts_bucket, fingerprint IN active_fps)`. **No `fmt.Sprintf`, no `%s` placeholders, no separate compile pass** — the previous `internal/modules/logs/querycompiler/` package was deleted. `logdetail` is fully `const` — single `(team_id, trace_id, span_id)` PREWHERE, no CTE. Repos hold queries only; service holds derivation: `log_facets` folds raw `(severity_bucket, service, host, pod, environment)` rows into per-dim top-N maps, `log_trends.Summary` consumes SQL-side `count() / countIf(severity_bucket >= 4) / countIf(severity_bucket = 3)` directly, `log_trends.Trend` does step-resolution / `formatBucket` / map-fold / sort. (Output models — `Log`, `LogRow`, `Cursor`, `Facets`, `Summary`, `TrendBucket`, `PageInfo`, `MapLogs`, `EncodeLogID`, etc. — stay in `internal/modules/logs/shared/models/`.)
- **Traces read path**: apm-style across the three filter-bearing submodules — `explorer` (POST `/traces/query`, list + facets/trend/summary include orchestrator), `span_query` (POST `/spans/query`), `errors` (GET `/errors/{groups,timeseries}` + per-service variants). Wire format is **flat typed JSON** for `explorer`/`span_query`; `errors` keeps its GET query-string shape (handler maps query params into `filter.Filters`). Typed shape and clause emitter live in [internal/modules/traces/filter/filter.go](internal/modules/traces/filter/filter.go) (`filter.Filters` + `filter.AttrFilter` + `Validate()` + `BuildClauses(f)`). Each filter-bearing repo method takes `filter.Filters` directly and calls **one shared `filter.BuildClauses(f) → (resourceWhere, where, args)`** with **stable bind names** (`services` / `excServices` / `httpStatuses` / `peerServices` etc.) so identical predicate sets produce byte-identical SQL — CH plan cache can hit. Inline `WITH active_fps AS (SELECT DISTINCT fingerprint FROM observability.spans_resource PREWHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd <resourceWhere>)` CTE in every query; main scan PREWHEREs `(team_id, ts_bucket, fingerprint IN active_fps)`. **No `fmt.Sprintf`, no `%s`, no separate compile pass** — `internal/modules/traces/querycompiler/`, `internal/modules/traces/shared/resource/`, `internal/modules/traces/shared/traceidmatch/`, `internal/modules/traces/shared/rootspan/`, and `internal/modules/traces/shared/models/` were all deleted (the last was just `filters.go` and migrated to the `filter` package). `latency` was moved to `internal/modules/services/latency/` (it's a service-domain panel, not a trace-ID-scoped concern) and rewritten apm-style with `quantileTiming` against raw spans.
- **Traces trace-ID-scoped read paths** (`tracedetail`, `trace_paths`, `trace_servicemap`, `trace_shape`, `trace_suggest`): apm-style point lookups. Each query is `const query = `…``, every method PREWHEREs `team_id = @teamID` (and additional PK slots where they're known: `span_id` for `GetSpanLogs`/`GetSpanAttributes`/`Subtree`'s start CTE; `service` + `name` for `GetRelatedTraces`). For trace-id-scoped queries the trace_id predicate stays in WHERE — the helper SQL fragment that handles OTLP empty/all-zero-hex/case-insensitive forms is **inlined as `traceIDMatchPredicate` const** at the bottom of each repository.go (with a per-file `traceIDArgs(teamID, traceID)` helper); the previous `internal/modules/traces/shared/traceidmatch/` and `internal/modules/traces/shared/rootspan/` helper packages were deleted (no remaining callers). Service owns derivation: `trace_paths` runs longest-path DAG traversal + error-chain walk, `trace_shape` builds flamegraph trees with self-time computation, `trace_servicemap` folds spans into service-map nodes/edges and groups errors, `tracedetail` parses the events array + exception fields into `SpanEvent`s (`splitEventRows` + `parseEventJSON` helpers), normalizes `db.statement` literals to `?` placeholders (`normalizeDBStatement`), and Subtree-walks the parent chain Go-side (`filterSubtree`). `tracedetail.GetRelatedTraces` is the lone window-scoped query among these — it gets the apm `WITH active_fps AS (... spans_resource PREWHERE team_id, ts_bucket, service ...)` CTE and main-scan PREWHERE on `(team_id, ts_bucket, fingerprint IN active_fps, service, name)`. `trace_suggest` is autocomplete: `service.IsScalarField` validates the canonical scalar name set up front (so the repo can string-concat the closed-set column expression safely), `service.fetch` dispatches `@`-prefix → `SuggestAttribute` vs scalar → `SuggestScalar`; both queries PREWHERE `(team_id, ts_bucket)`. **No `fmt.Sprintf`, no `%s`, no shared SQL-fragment helpers** across these 5 submodules.
- **Service topology read path** (`internal/modules/services/topology/`): apm-style. One route — `GET /api/v1/services/topology[?service=<focus>]` — backed by two queries that fan out via `errgroup`. Repository holds queries only; service holds derivation. `GetNodes` aggregates per-service RED counters (`count()`, `countIf(has_error OR toUInt16OrZero(response_status_code) >= 400)`) plus a fixed-bucket latency histogram (`Array(UInt64)` from a chain of `countIf(duration_nano …)` calls); `GetEdges` derives directed edges from `kind_string = 'Client'` spans where `service` and `peer_service` are non-empty and distinct, emitting the same RED + histogram shape per `(source, target)`. Both queries run inline `WITH active_fps AS (SELECT fingerprint FROM observability.spans_resource PREWHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd)` CTEs; main scan PREWHEREs `(team_id, ts_bucket, fingerprint IN active_fps)` for full PK granule pruning. **No quantile aggregate runs in CH** — the bucket array stays in lockstep with `latencyBucketBoundsMs` in `service.go` (`5/10/25/50/100/250/500/1000/2500/5000/10000/30000/+Inf` ms upper bounds), and `service.buildNodes`/`service.buildEdges` interpolate P50/P95/P99 Go-side via [quantile.FromHistogram](internal/shared/quantile/quantile.go). Service also derives error-rate (`error_count / count`), classifies node health into `healthy`/`degraded`/`unhealthy` via `unhealthyErrorRate=0.05` / `degradedErrorRate=0.01` thresholds, calls `addMissingEdgeNodes` to inject pure-CLIENT producers as nodes, and applies `filterNeighborhood` for the optional 1-hop focus filter. Repo-local `spanArgs(teamID, startMs, endMs)` and `spanBucketBounds(startMs, endMs)` helpers mirror the per-module pattern in `services/latency`. **No `fmt.Sprintf`, no phantom-rollup column references** (`latency_ms_digest`, `client_service`, `server_service` were referenced in the old query against non-existent rollup tables — same broken-against-non-existent-rollup problem `services/latency` had pre-refactor).
- **Metrics explorer read path** (`internal/modules/metrics/`): apm-style, structured as a sub-module like logs/traces. The `explorer` submodule (`internal/modules/metrics/explorer/`) owns the full Module (handler/service/repo/dto/models/module.go); `internal/modules/metrics/filter/` is a sibling subpackage holding the typed shape and clause-emitter consumed by the repo. Three routes — `GET /metrics/names` (autocomplete), `GET /metrics/:metricName/tags` (key+value autocomplete), `POST /metrics/explorer/query` (main aggregation). Wire format keeps the **dynamic** `where: [{key, operator, value}]` shape because metrics is a generic open-dimension explorer (any tag, any operator) — promoting some keys to typed top-level fields would create an asymmetric API. Typed `filter.Filters` lives in the [internal/modules/metrics/filter/](internal/modules/metrics/filter/) subpackage (deviates from logs/traces' per-module-helper-at-the-bottom-of-repository.go convention — chosen for metrics so canonicalization, sanitization, and clause emission share one home). The `filter` package owns `Filters` + `TagFilter` + `Validate()` (clamps to ≤30 days, validates aggregation against `validAggregations`), `Canonical()` (alias map: `service.name`→`service`, `host.name`→`host`, `deployment.environment`→`environment`, `k8s.namespace.name`→`k8s_namespace`), `ResourceColumn()` (4 flat columns on `metrics_resource`), `AttrColumn()` (sanitized `attributes.\`<key>\`::String` JSON-path on raw `metrics`), `GroupByColumn()`, `SanitizeKey()`, and `BuildClauses(f) → (resourceWhere, where, args)` which partitions `f.Tags` into the resource-CTE fragment and the row-side WHERE fragment. **Stable bind names** for the 4 canonical resource keys (`services` / `excServices` / `hosts` / `excHosts` / `environments` / `excEnvironments` / `k8sNamespaces` / `excK8sNamespaces`); arbitrary attribute filters use indexed `mf0`/`mf1`/… binds. Resource clauses are emitted in **fixed order** so identical filter shapes produce byte-identical SQL even when the input list ordering varies — CH plan cache can hit. **Every query PREWHEREs the metrics PK**: `ListMetricNames` is a single statement with a `WITH names AS (SELECT DISTINCT metric_name FROM observability.metrics_resource PREWHERE team_id = @teamID AND ts_bucket BETWEEN @bucketStart AND @bucketEnd WHERE metric_name ILIKE @search)` candidate-narrowing CTE, then PREWHEREs raw `metrics` on `(team_id, ts_bucket_hour, metric_name IN names)` for type/unit/description lookup; `ListTagKeys` PREWHEREs raw metrics on `(team_id, ts_bucket_hour, metric_name)`; `ListTagValues` resource path PREWHEREs `metrics_resource` on `(team_id, ts_bucket, metric_name)`, data-point path PREWHEREs raw metrics on `(team_id, ts_bucket_hour, metric_name)`; `QueryTimeseries` runs an inline `WITH active_fps AS (... metrics_resource PREWHERE ...)` CTE then PREWHEREs raw metrics on `(team_id, ts_bucket_hour, fingerprint IN active_fps, metric_name)`. One round-trip per filtered query — the previous `internal/modules/metrics/shared/resource/` preflight resolver was deleted. **No `fmt.Sprintf`, no `%s`** — every query is a `const query = \`...\`` (or string-concat with the closed set of resource columns / sanitized AttrColumn for `ListTagValues`). Aggregation expressions (`avg`/`sum`/`min`/`max`/`count`/`rate`/`p50`-`p99`) emit via `buildAggExpr`; the rate divisor uses `strconv.FormatInt`, not `fmt.Sprintf`. Repo-local `metricBucketBounds(startMs, endMs)` and `metricArgs(f)` helpers produce hour-aligned `[bucketStart, bucketEnd)` and the 6 base bind values. Service-layer `convertFEQuery` maps the FE filter shape into `filter.Filters`, validates, hands off to repo; `buildColumnarResult` folds flat per-bucket rows into the columnar `{timestamps, series}` wire shape the frontend expects.
- **Spans schema (`observability.signoz_index_v3`)**: hot resource/span attributes are stored as JSON typed-path subcolumns inside `attributes JSON(...)` (e.g. `attributes.\`service.name\``) — column-fast reads, no JSON parse per row. Reader-facing names (`service_name`, `host_name`, `pod_name`, `service_version`, `deployment_environment`, `peer_service`, `db_system`, `db_name`, `db_statement`, `http_route`, `attr_exception_type`) are zero-cost `ALIAS` columns. `resource_fingerprint` is a real top-level column written by the mapper. Resource-side bloom filters (service/host/pod/version/env) were removed because the resolver pre-filters those into `resource_fingerprint IN (...)`. **No rollup tables exist** — `spans_red_rollup`, `spans_peer_rollup`, `spans_by_version_rollup`, `spans_error_fingerprint`, `signoz_index_v3_rollup_*`, `signoz_index_v3_host_rollup_*`, `db_histograms_rollup`, `root_spans_index`, `traces_index`, `observability.resources` were planned but never built; readers go to raw `signoz_index_v3` and rely on resolver narrowing.

## Engineering principles

- **Module Architecture**: Strict 6-file pattern (`handler.go`, `service.go`, `repository.go`, `module.go`, `dto.go`, `models.go`). All repository methods must stay in `repository.go`.
- **SOLID & DRY**: Factor shared behavior when a pattern appears more than once.
- **Quality**: Leave the code clearer or simpler with every change.
- **No unsolicited tests**: Do not add tests unless explicitly asked.
